Introduction

What is KEDA?

Autoscaling = Kubernetes automatically adds or removes running copies (pods) of your app. More traffic → more pods. Less traffic → fewer pods. You pay for what you actually use.
The problem with built-in HPA

The standard Kubernetes autoscaler (HPA) only understands CPU and Memory. These are lagging indicators — your users feel the slowdown before Kubernetes acts.

  • 10,000 messages in a queue? HPA has no idea
  • Response time at 3 seconds? HPA doesn't see it
  • 03:00 with zero users? Still running 5 pods, still paying
  • Want to scale to zero? HPA minimum is 1
KEDA fills the gap

KEDA (Kubernetes Event-Driven Autoscaling) was created by Microsoft and Red Hat. It connects Kubernetes to 77 external event sources and lets your app scale on the signal that actually reflects real demand.

KEDA is non-destructive — it doesn't replace the HPA or any Kubernetes component. It sits alongside them and feeds them better data.

Three components inside your cluster
ComponentWhat it does
keda-operatorWatches ScaledObjects, owns the 0 ↔ 1 scaling step, manages HPA lifecycle
keda-metrics-apiserverExposes external metric values to the HPA — bridges external world to Kubernetes
keda-admission-webhooksValidates config at apply time — catches mistakes before they cause runtime failures
Two-phase scaling lifecycle

Phase 1 — 0 to 1: keda-operator handles this directly. When events arrive and there are zero pods, KEDA brings the first pod up.

Phase 2 — 1 to N: The HPA takes over, using metrics provided by keda-metrics-apiserver to drive replica count up or down.

Scale to zero: Only KEDA can do this. The HPA's minimum is 1. KEDA's keda-operator handles the final 1 → 0 step when all triggers go idle.

77 built-in scalers covering Kafka, SQS, RabbitMQ, Prometheus, Datadog, Dynatrace, Redis, PostgreSQL, Cron, GitHub Actions, and many more. Plus a gRPC interface to write your own.
Introduction

HPA vs KEDA — what actually changes

HPA (Horizontal Pod Autoscaler) is built into Kubernetes. KEDA extends it — you keep the HPA, you just give it more signals to act on. Selectively applied: only workloads with a ScaledObject are affected.

Cron Scaler

Cron Scaler — scale by the clock

Cron is a time-based scheduler — like a calendar alarm for your app. The KEDA cron trigger lets you define exactly when replicas should go up and when they should come back down.
Real-world use cases
  • Pre-warm before office hours: start pods at 08:45 so the first employee at 09:00 hits a warm app, not a cold start
  • Save cost overnight: scale to zero after 18:00, bring back at 08:45 — pay nothing for 15 hours
  • Weekend minimum: keep 1 pod on weekends for health checks, scale to 5 on Monday 08:00
  • Combine with metric triggers: cron sets the minimum floor, Prometheus or Dynatrace drives the actual count during business hours
Cron sets a floor, never a ceiling. If Prometheus wants 8 replicas and cron wants 2, KEDA picks 8. The cron trigger never prevents scale-up.
Always set timezone by name (e.g., Europe/Stockholm), not as a UTC offset. Named timezones automatically handle daylight saving time. Without a timezone, cron defaults to UTC — your 09:00 schedule fires at the wrong local time. 
- type: cron
  metadata:
    timezone: "Europe/Stockholm"
    # ↑ Named timezone — handles daylight saving
    #   automatically. Always required.
    #   Examples: "UTC", "America/New_York",
    #             "Asia/Kolkata", "Europe/Berlin"

    start: "0 9 * * 1-5"
    # ↑ Standard 5-field cron expression:
    #   minute  hour  day  month  weekday
    #       0     9    *     *    1-5
    #   = 09:00, every Monday–Friday
    #   Trigger becomes ACTIVE at this time.

    end: "0 18 * * 1-5"
    # ↑ 18:00, Monday–Friday.
    #   Trigger becomes INACTIVE.
    #   If this is the last active trigger,
    #   cooldownPeriod begins and replicas
    #   fall back to minReplicaCount.

    desiredReplicas: "2"
    # ↑ While active: ensure at least 2 replicas.
    #   This is a FLOOR, not a ceiling.
    #   A Prometheus trigger seeing heavy traffic
    #   can still push this to 10 — cron won't
    #   limit it.
Prometheus Scaler

Prometheus Scaler — scale on any metric

Prometheus is an open-source metrics database that continuously collects numbers from your applications — connection counts, request rates, queue sizes, error counts. KEDA queries it using PromQL (the Prometheus query language) and uses the result to drive scaling decisions.
What you can scale on
SignalExample PromQL
Active connectionssum(nginx_connections_active)
HTTP requests/secsum(rate(http_requests_total[2m]))
Messages in queuerabbitmq_queue_messages_ready
P95 response timehistogram_quantile(0.95, rate(...))
Error raterate(http_errors_total[2m])
AWS AMP (Amazon Managed Prometheus): the managed scraper doesn't add a namespace label to scraped metrics. Filtering by namespace returns zero results. Use the metric name to identify your app instead. 
- type: prometheus
  authenticationRef:
    name: aws-amp
    kind: ClusterTriggerAuthentication
    # ↑ Separate auth object — credentials are
    #   NEVER written in the ScaledObject YAML.
    #   This one uses IRSA (IAM Roles for Service
    #   Accounts) — no static keys anywhere.

  metadata:
    serverAddress: "${AMP_WORKSPACE_URL}"
    # ↑ The Prometheus server to query.
    #   ${...} is substituted by Flux at deploy
    #   time — the URL never lives in git.

    metricName: nginx_connections_active
    # ↑ A name KEDA uses internally to identify
    #   this metric in the HPA object. Must be
    #   unique within the ScaledObject.

    query: "sum(nginx_connections_active)"
    # ↑ PromQL sent to Prometheus every
    #   pollingInterval seconds.
    #   sum() adds values from all pods:
    #   pod-A=60, pod-B=70 → total=130

    threshold:           "20"
    # ↑ Target connections PER replica.
    #   Formula: ceil(130 / 20) = 7 replicas.
    #   "I want each replica to handle ≤20."

    activationThreshold: "10"
    # ↑ Trigger stays INACTIVE below 10.
    #   Browsers hold idle "keep-alive" TCP
    #   connections (3–6) even when not sending
    #   requests. Without this, those idle conns
    #   would falsely activate scaling.

    awsRegion: "eu-north-1"
    # ↑ Required for AWS SigV4 request signing.
Dynatrace Scaler

Dynatrace Scaler — scale on observability signals

Dynatrace continuously monitors your services end-to-end — response times, error rates, throughput, infrastructure metrics. The KEDA Dynatrace scaler queries the Metrics v2 API using a structured selector expression, letting you scale on anything Dynatrace can observe.
Reading a Metrics v2 selector — left to right
nginx_connections_active      ← 1. metric name
  :filter(and(
    eq(k8s.namespace.name,keda-testing)
  ))                            ← 2. only this namespace
  :sum                          ← 3. add up all matching pods
  :fold                         ← 4. collapse time window to ONE number
                                     (KEDA needs a single value to compare
                                     against threshold)
API host: must be https://abc.live.dynatrace.com
— not apps.dynatrace.com (Metrics v2 API not available there → 404)
— must include https:// (missing scheme causes "unsupported protocol" error)
Always set from: "now-2m". The default query window is 2 hours. A 3-minute traffic spike averaged over 2 hours is mathematically invisible — :fold returns a near-zero average and your trigger never fires. Use a short window that matches your expected load pattern. 
- type: dynatrace
  authenticationRef:
    name:  keda-trigger-auth-dynatrace
    kind:  ClusterTriggerAuthentication
    # ↑ The DT API token lives in an encrypted
    #   Kubernetes Secret — never in this file.
    #   Flow: plaintext token → kubeseal encrypt
    #   → commit YAML to git (safe, encrypted)
    #   → Flux deploys → controller decrypts
    #   → K8s Secret in cluster only.

  metadata:
    metricSelector:
      "nginx_connections_active:filter(and(
       eq(k8s.namespace.name,keda-testing)))
       :sum:fold"
    # ↑ Full Dynatrace Metrics v2 selector.
    #   Read left to right:
    #   metric → namespace filter → sum → fold

    from: "now-2m"
    # ↑ Query window: last 2 minutes only.
    #   CRITICAL — do not omit this field.
    #   Default is now-2h which dilutes spikes.

    threshold:           "10"
    # ↑ Target per replica: ceil(metric / 10).
    #   50 connections → 5 replicas.

    activationThreshold: "7"
    # ↑ nginx keeps 3–6 idle keep-alive conns.
    #   Setting this to 7 means those idle
    #   background connections never activate
    #   the trigger — only real traffic does.
Live Demo

The ScaledObject — every field explained

apiVersion: keda.sh/v1alpha1
kind:       ScaledObject    ← KEDA custom resource
metadata:
  name:      nginx-keda-sample
  namespace: keda-testing
spec:
  scaleTargetRef:
    name: nginx-keda-sample    ① exact Deployment name — must match
  minReplicaCount: 1          ② minimum pods (default: 0)
  maxReplicaCount: 10         ③ hard cap passed to HPA (default: 100)
  pollingInterval: 15         ④ check triggers every 15 s (default: 30)
  cooldownPeriod:  60         ⑤ wait before scale-to-zero (default: 300)
  advanced:
    horizontalPodAutoscalerConfig:
      behavior:
        scaleDown:
          stabilizationWindowSeconds: 60  ⑥ HPA scale-down delay
  triggers:
    - type: cron               ⑦ replica floor: 18:00–06:00
      metadata:
        timezone:        "Europe/Stockholm"
        start:           "0 18 * * *"
        end:             "0 6 * * *"
        desiredReplicas: "1"
    - type: prometheus          ⑧ scale on connections from AWS AMP
      authenticationRef:
        name: aws-amp
        kind: ClusterTriggerAuthentication
      metadata:
        serverAddress:       "${AMP_WORKSPACE_URL}"
        metricName:          nginx_connections_active
        query:               "sum(nginx_connections_active)"
        threshold:           "20"
        activationThreshold: "10"
        awsRegion:           "eu-north-1"
    - type: dynatrace           ⑨ scale on connections from Dynatrace
      authenticationRef:
        name: keda-trigger-auth-dynatrace
        kind: ClusterTriggerAuthentication
      metadata:
        metricSelector:
          "nginx_connections_active:filter(and(
           eq(k8s.namespace.name,keda-testing))):sum:fold"
        from:                "now-2m"
        threshold:           "10"
        activationThreshold: "7"
① scaleTargetRef — what to scale

Must exactly match the Deployment name. KEDA silently fails to reconcile if this is wrong — the admission webhook will catch it at kubectl apply.

② min=1   ③ max=10 — guardrails

min=1: always one warm pod — no cold-start delay for users. max=10: hard safety cap. Default is 100 — a misbehaving metric can spawn 100 pods without this.

④ pollingInterval · ⑤ cooldownPeriod · ⑥ stabilization

pollingInterval: 15 — KEDA checks every 15 s (default 30 s).
cooldownPeriod: 60 — after all triggers go idle, wait 60 s before scaling to zero (default 300 s — very slow without override).
stabilizationWindowSeconds: 60 — HPA holds the highest recent replica count for 60 s before scaling down (default 300 s — must set both to avoid staying at peak for 5 minutes).

⑦ ⑧ ⑨ Three triggers — OR logic, MAX wins

All three run in parallel, evaluated every pollingInterval. KEDA calculates desired replicas from each active trigger and forwards the maximum to the HPA.
Example: cron=1, Prometheus=5, Dynatrace=8 → HPA target = 8.

desiredReplicas = ceil( metric ÷ threshold )
How it works

How the scaling math works

desiredReplicas = ceil( metricValue ÷ threshold )  — same formula Kubernetes HPA uses
Phase 1 — Is this trigger even awake?

Every trigger has an activationThreshold. The trigger is INACTIVE (ignored completely) until the metric exceeds this value. This filters out idle background noise.

# Idle: 4 keep-alive connections, activationThreshold=7
4 ≤ 7  → trigger INACTIVE → stays at minReplicaCount=1

# Load arrives: 100 connections
100 > 7 → trigger ACTIVE → run the formula

# Important: if minReplicaCount ≥ 1, the trigger is
# always considered active — activationThreshold is
# only meaningful when minReplicaCount = 0.
Phase 2 — How many replicas?
# 100 connections, threshold=10 per replica
ceil( 100 ÷ 10 ) = 10 replicas

# "I want ≤10 connections per replica.
#  With 100 total I need 10 pods."

# Multiple triggers — KEDA picks MAX:
cron=1, prometheus=5, dynatrace=8
→ HPA target = 8

Why DT fires before AMP

Dynatrace OneAgent streams metrics continuously — KEDA sees the spike within one pollingInterval (15 s). AWS AMP scrapes on a fixed interval (typically 60 s) — it may not have sampled since the load started. This is why having both scalers provides faster reaction: whichever sees the load first wins.

Security

Keeping credentials secure

KEDA needs to call external APIs (Prometheus, Dynatrace) using credentials. Never put passwords, tokens, or API keys directly in the ScaledObject YAML — that file lives in git and is visible to anyone with repo access. KEDA's TriggerAuthentication resource separates credentials from scaling config entirely.
AWS Prometheus (AMP) — IRSA

IRSA = IAM Roles for Service Accounts. The KEDA pod gets an AWS IAM identity. AWS automatically injects short-lived credentials (15-minute expiry) — there is no password to write down, store, rotate, or leak.

apiVersion: keda.sh/v1alpha1
kind:       ClusterTriggerAuthentication
metadata:
  name: aws-amp
spec:
  podIdentity:
    provider: aws
# That's the entire config.
# AWS provides creds from the pod's
# service account annotation:
#   eks.amazonaws.com/role-arn: arn:...
# Rotates automatically every 15 minutes.
# Nothing to store, nothing to rotate,
# nothing to leak.
Dynatrace — SealedSecret

SealedSecrets (Bitnami) encrypt a Kubernetes Secret with the cluster's public key. The encrypted YAML is safe to commit to git — only that specific cluster can decrypt it.

# The flow from token to KEDA:
1. DT API token (plaintext, never committed)
   ↓  kubeseal --controller-namespace flux-system
2. SealedSecret YAML (encrypted) → commit to git ✓
   ↓  Flux deploys to cluster
3. sealed-secrets controller decrypts it
   ↓  creates a real Kubernetes Secret (cluster only)
4. ClusterTriggerAuthentication reads the Secret
   ↓
5. KEDA uses token to call Dynatrace API

# ClusterTriggerAuthentication spec:
spec:
  secretTargetRef:
    - parameter: token
      name: dynatrace-token  key: token
    - parameter: host
      name: dynatrace-token  key: host
Your Checklist

What you own as the developer

1. Choose the right scaling signal
Your workloadScale onWhy
Processes a message queueQueue depthDirect measure of work backlog
Serves HTTP requestsRequest rateLeads CPU — react before slowdown
Has a latency SLOP95 response timeScale on user experience directly
Has predictable hoursCron + metricPre-warm; metric drives actual count
CPU-bound computeStandard HPA (CPU)CPU is the actual bottleneck here
Avoid CPU as the primary signal for event-driven workloads. CPU spikes happen after requests are already queuing. A leading signal (queue depth, connection count) lets KEDA add pods before users notice.
2. Set safe replica guardrails
  • min=0: only for async batch jobs. Cold start (image pull + app init) takes 15–60 s — users on a web API will notice.
  • min=1: for user-facing APIs — always one warm pod.
  • min=2: for HA — one pod can restart or be evicted without any downtime.
  • max — always set explicitly. Default is 100. Calculate: max × connections/pod ≤ DB max_connections. Same logic for external API rate limits and Kafka partitions.
3. Know your downstream impact
  • Databases: 20 pods × 10 conn/pod = 200 connections — check max_connections
  • External APIs: more pods = more parallel calls = easier to hit rate limits
  • Kafka: consumer pods ≤ partition count; extra pods sit idle
  • Stateful apps (WebSocket, in-flight DB transactions): add a preStop lifecycle hook so pods drain gracefully before terminating — otherwise connections drop mid-request
